# 社内打ち合わせ予約システム 開発指示書

---

## 1. プロジェクト概要

### 1.1 背景と目的

本プロジェクトは、社内における月次定例打ち合わせの日程調整・予約管理を効率化するための社内向けWebアプリケーションを開発することを目的とする。現状、メールやチャットツールを介した日程調整に伴う工数・コミュニケーションコストの削減を目指す。

### 1.2 利用対象者

| ロール | 説明 |
|---|---|
| **マネジメント部** | 打ち合わせ可能な候補日程を登録・管理する側 |
| **エンジニア** | 提示された候補日程から希望日を選択・予約する側 |

### 1.3 利用シナリオ

- 月1回の頻度で開催される社内定例打ち合わせを対象とする
- マネジメント部が翌月の打ち合わせ候補日程を事前に登録する
- エンジニアが候補日程の中から都合の良い日時を選んで予約を確定する
- 予約状況は双方がリアルタイムで確認できる

### 1.4 スコープ

- **対象：** 社内ユーザーのみ（外部公開なし）
- **対象会議：** 月次定例打ち合わせ（1対1 または グループ形式）
- **対象外：** 外部顧客との予約、会議室リソース管理（将来拡張として検討可）

## 2. 機能要件

### 2.1 認証・ユーザー管理

- **FR-01:** ユーザーはメールアドレスとパスワードによるログインができること
- **FR-02:** ユーザーには「マネジメント部」「エンジニア」のロールが付与されること
- **FR-03:** ロールに応じて利用できる機能が制限されること（ロールベースアクセス制御）
- **FR-04:** アカウントの新規登録は管理者（情報システム部）のみが行えること。ユーザーによる自己登録は不可とする
- **FR-05:** 管理者がアカウントを作成した際、以下の流れで初回ログインを案内すること
  1. 登録したメールアドレスへ「パスワード設定リンク」を自動送信する
  2. ユーザーはリンクから自分のパスワードを設定してログインする
  3. リンクの有効期限は24時間とし、期限切れの場合は管理者が再送できること
- **FR-06:** 管理者はユーザーの追加・編集・削除・ロール変更ができること

### 2.2 候補日程管理（マネジメント部）

- **FR-07:** マネジメント部は打ち合わせの候補日時（複数）を登録できること
- **FR-08:** 候補日程には以下の情報を含めること
  - 日付・開始時刻・終了時刻
  - 打ち合わせタイトル / テーマ
  - 定員（最大参加人数）
  - 備考・補足説明
- **FR-09:** 登録済みの候補日程を編集・削除できること
- **FR-10:** 既に予約が入っている候補日程を削除する場合は、予約者へ通知する機能を持つこと
- **FR-11:** 候補日程の公開・非公開を切り替えられること（下書き状態）

### 2.3 日程予約（エンジニア）

- **FR-12:** エンジニアは公開済みの候補日程一覧を閲覧できること
- **FR-13:** エンジニアは希望する候補日程を選択して予約を申し込めること
- **FR-14:** 予約時には**面談方法**を以下の選択肢から選択できること（必須項目）
  - `対面` ： 社内の会議室など物理的な場所での実施
  - `Web` ： オンライン会議ツール（Google Meet）での実施
- **FR-15:** 面談方法に `Web` を選択した場合、予約確定時にシステムが自動的にGoogle MeetのURLを発行し、予約者（エンジニア）およびマネジメント部の双方へメールで通知すること
- **FR-16:** 発行されたGoogle Meet URLは「マイ予約一覧画面」および予約確認メール内に表示されること
- **FR-17:** 予約時には任意でコメント（議題・質問事項など）を入力できること
- **FR-18:** 同一ユーザーが同一スロットに重複して予約できないよう制御すること
- **FR-19:** 定員に達したスロットは予約不可状態として表示すること（満員表示）
- **FR-20:** エンジニアは自分の予約をキャンセルできること（開催直前のキャンセル制限は設定で対応）

### 2.4 予約状況管理・承認フロー（マネジメント部）

- **FR-21:** マネジメント部は各候補日程に対する予約者一覧（氏名・面談方法・コメント・承認ステータス）を確認できること
- **FR-22:** エンジニアが予約を申し込んだ時点では「承認待ち」ステータスで保存され、仮予約状態となること
- **FR-23:** マネジメント部は予約一覧画面から各予約に対して「承認」または「却下」を実行できること
- **FR-24:** 承認操作を行った際、以下の処理がシステムにより自動で実行されること
  1. DBの予約ステータスを「承認済み」に更新する
  2. エンジニアへ承認確定メールを送信する
  3. Google Calendar API を呼び出し、エンジニア・マネジメント部双方のGoogleカレンダーに予定を自動追加する
  4. 面談方法が `Web` の場合：Google Meet URLを発行し、カレンダー予定および確定メールに含める
- **FR-25:** 却下操作を行った際、以下の処理がシステムにより自動で実行されること
  1. DBの予約ステータスを「却下」に更新する
  2. エンジニアへ却下通知メールを送信する（却下理由を任意で入力可能）
- **FR-26:** 予約をCSV形式でエクスポートできること（ステータス・面談方法・Google Meet URLを含む）

### 2.5 通知機能

- **FR-27:** 以下のタイミングでメール通知が送信されること
  - エンジニアが予約を申し込んだとき（マネジメント部へ承認依頼メールを送信）
  - マネジメント部が承認したとき（エンジニアへ確定メールを送信。面談方法が `Web` の場合はGoogle Meet URLを記載）
  - マネジメント部が却下したとき（エンジニアへ却下通知メールを送信）
  - エンジニアが予約をキャンセルしたとき（予約者本人・マネジメント部へ）
  - 候補日程が更新・削除されたとき（予約済みエンジニアへ）
  - 打ち合わせ日時の前日（リマインダー）
    - 面談方法が `Web` の場合、リマインダーメールにもGoogle Meet URLを再掲すること

### 2.6 カレンダー連携

- **FR-28:** マネジメント部が承認を実行した時点で、Google Calendar API を通じてエンジニア・マネジメント部双方のGoogleカレンダーに予定を自動追加すること
  - カレンダーへの書き込みにはOAuth 2.0によるユーザー認可が必要なため、初回ログイン時にGoogleアカウントとの連携設定を行うこと
  - 面談方法が `Web` の場合、カレンダー予定の場所欄にGoogle Meet URLを含めること
  - API呼び出しが失敗した場合はエラーをログに記録し、管理者へ通知するとともに手動追加リンクをメールで案内すること

## 3. UI/UX 要件

### 3.1 画面構成

```
[共通]
├── ログイン画面
├── ダッシュボード（ホーム）
│
[マネジメント部]
├── 候補日程管理画面
│   ├── 候補日程一覧
│   ├── 候補日程登録／編集フォーム
│   └── 予約者確認画面（スロット別）
├── 予約状況サマリー画面
└── ユーザー管理画面（管理者のみ）
│
[エンジニア]
├── 候補日程一覧・予約画面
├── マイ予約一覧画面
└── プロフィール設定画面
```

### 3.2 ダッシュボード

- ログイン直後に表示されるホーム画面
- マネジメント部：未確認の新規予約件数、直近の打ち合わせ一覧を表示
- エンジニア：自分の予約済み打ち合わせ、新しく公開された候補日程を表示
- 未対応アクション（予約していない公開中スロットなど）をバッジで通知

### 3.3 候補日程一覧・予約画面（エンジニア向け）

- カレンダービューとリストビューを切り替えられること
- 残り定員数を視覚的に表示（プログレスバー等）
- 予約済み・満員・キャンセル済みなど、ステータスを色やアイコンで区別
- 候補日程のフィルタリング・ソート機能（日付、テーマ等）

### 3.4 フォーム設計

- 必須項目・任意項目を明確に区別する
- バリデーションはリアルタイムで行い、エラーメッセージはフィールドの直下に表示
- 予約確定前に確認ダイアログを表示する
- 操作完了後はトースト通知でフィードバックを行う

### 3.5 レスポンシブ対応

- PC・タブレット・スマートフォンの各デバイスで適切に表示されること
- モバイルでの操作を考慮し、ボタンやタップターゲットは十分なサイズを確保する（推奨: 44px以上）

### 3.6 アクセシビリティ

- WAI-ARIA 対応（スクリーンリーダー利用者への配慮）
- カラーコントラスト比は WCAG 2.1 AA 基準を満たすこと
- キーボード操作のみで全機能が利用できること

## 4. 技術要件

### 4.1 推奨技術スタック

#### フロントエンド

| 項目 | 推奨技術 | 理由 |
|---|---|---|
| フレームワーク | React / Next.js | コンポーネント再利用性・SSR対応・エコシステムの豊富さ |
| 言語 | TypeScript | 型安全による開発効率・保守性の向上 |
| UIライブラリ | shadcn/ui または MUI | デザイン統一性・アクセシビリティ対応 |
| 状態管理 | Zustand / React Query | 軽量・シンプル、サーバーキャッシュ管理 |
| カレンダー | FullCalendar または react-big-calendar | カレンダーUI の実装コスト削減 |
| フォーム | React Hook Form + Zod | バリデーション統合・型安全 |

#### バックエンド

| 項目 | 推奨技術 | 理由 |
|---|---|---|
| フレームワーク | Node.js (Express / Fastify) または Python (FastAPI) | REST API の構築容易性・社内スキルセットに応じて選択 |
| 言語 | TypeScript（Node.js）または Python | 型安全・保守性 |
| ORM | Prisma（Node.js）または SQLAlchemy（Python）| DBスキーマ管理・型安全なクエリ |
| 認証 | JWT + Refresh Token / NextAuth.js | セキュアなセッション管理 |
| メール送信 | Nodemailer / SendGrid | 通知メール送信 |
| Google Meet連携 | Google Calendar API（`conferenceData`） | Web面談時のMeet URL自動発行 |

#### データベース

| 項目 | 推奨技術 |
|---|---|
| RDBMS | PostgreSQL（推奨）または MySQL |
| キャッシュ | Redis（セッション管理・通知キュー用） |

#### インフラ・DevOps

| 項目 | 推奨技術 |
|---|---|
| ホスティング | AWS / GCP / Azure（社内インフラに準拠） |
| コンテナ | Docker + Docker Compose |
| CI/CD | GitHub Actions |
| バージョン管理 | Git / GitHub または GitLab |

### 4.2 APIの設計方針

- RESTful API 設計に従うこと
- エンドポイントはリソース指向で命名すること（例: `GET /api/slots`, `POST /api/reservations`）
- APIバージョニングを導入すること（例: `/api/v1/...`）
- レスポンスは統一フォーマット（`{ data, error, meta }`）を使用すること

### 4.3 セキュリティ要件

- **SE-01:** パスワードはbcrypt（コストファクター12以上）でハッシュ化して保存すること
- **SE-02:** SQLインジェクション・XSS・CSRFへの対策を実施すること
- **SE-03:** APIエンドポイントはすべて認証・認可チェックを行うこと
- **SE-04:** HTTPS通信を必須とすること
- **SE-05:** 機密情報（APIキー・DB接続情報）は環境変数で管理し、リポジトリにコミットしないこと
- **SE-06:** 入力値はサーバーサイドでバリデーションを行うこと

### 4.4 パフォーマンス要件

- 画面初期表示は3秒以内（LCP）
- APIレスポンスは通常操作で500ms以内
- 同時接続ユーザー数：最大100名程度を想定

## 5. 制約事項

### 5.1 開発上の制約

| No. | 制約内容 |
|---|---|
| C-01 | 社内ネットワーク内からのみアクセス可能とする（VPN対応またはイントラネット限定） |
| C-02 | 外部への個人情報送信は行わないこと。メール通知は社内メールサーバー経由を原則とする |
| C-03 | 既存の社内認証基盤（LDAP / Active Directory等）がある場合は連携を検討すること |
| C-04 | データは社内サーバーまたは承認済みクラウド環境に保存すること |
| C-05 | OSSライセンスはMIT / Apache 2.0 / BSDに限定し、GPLライセンスは使用禁止 |

### 5.2 運用上の制約

| No. | 制約内容 |
|---|---|
| C-06 | システム管理者は情報システム部が担当する |
| C-07 | バックアップは日次で取得し、7世代以上を保持すること |
| C-08 | 障害発生時のRTO（目標復旧時間）は翌営業日以内とする |
| C-09 | ユーザーデータの保持期間は1年間とし、期限超過データは削除または匿名化すること |

### 5.3 開発プロセスの前提条件

| No. | 前提条件 |
|---|---|
| C-10 | 開発は Gitフローに従ってブランチ管理を行うこと（`main` / `develop` / `feature/*`） |
| C-11 | プルリクエストには最低1名のレビュワーによる承認を必須とする |
| C-12 | 主要機能にはユニットテスト・統合テストを作成し、カバレッジ70%以上を維持すること |
| C-13 | コーディング規約（ESLint / Prettier 等）に準拠すること |
| C-14 | READMEにローカル環境構築手順・環境変数一覧を必ず記載すること |

### 5.4 フェーズ分割（推奨）

本システムは以下のフェーズで段階的にリリースすることを推奨する。

```
Phase 1（MVP）
├── ユーザー認証・ロール管理（Googleアカウント連携含む）
├── 候補日程の登録・編集・削除
├── エンジニアによる日程予約（面談方法選択：対面 / Web）
└── 予約状況の確認

Phase 2
├── 承認フロー（承認待ち → 承認 / 却下）
├── 承認時のGoogleカレンダー自動追加
├── 承認時のGoogle Meet URL自動発行（Web面談）
└── メール通知機能（申込・承認・却下・リマインダー）

Phase 3（拡張）
├── CSV エクスポート
├── 管理ダッシュボード強化
└── 会議室リソース管理
```

## 付録：用語定義

| 用語 | 定義 |
|---|---|
| **スロット** | マネジメント部が登録した特定の日時・時間帯の打ち合わせ枠 |
| **候補日程** | マネジメント部が提示する、打ち合わせ可能な日時の選択肢群 |
| **予約** | エンジニアが特定のスロットに対して申し込んだ状態 |
| **定員** | 1スロットに参加できる最大人数 |
| **ロール** | システム内での権限区分（マネジメント部 / エンジニア / 管理者） |
